### **E2EE Chat Application: Technical Design Document (v0.4)**

#### **1. Overview**

This document outlines the design for an end-to-end encrypted (E2EE) chat application. The system ensures that only the communicating users can read messages.

The design uses a "separation of concerns" model:
*   **Authentication (Server-Side)**: A standard JWT mechanism controls which users can access the server's resources (e.g., uploading or downloading messages).
*   **Privacy (End-to-End)**: A hybrid cryptographic system ensures message content is private and authentic, completely opaque to the server.

#### **2. Actors**

*   **Client**: The user-facing application responsible for key management, encryption, decryption, and session (JWT) management.
*   **Server**: A backend service that authenticates users, stores, and forwards encrypted data blobs. It has zero visibility into message content.

#### **3. Cryptographic Flow**

1.  **Key Generation**: Each user generates a long-term **RSA-2048** key pair (`SK`, `PK`). The private key (`SK`) never leaves the device.

2.  **User & Key Registration**: A challenge-response mechanism proves the user owns the private key corresponding to the public key they are registering.

3.  **Session Establishment**: A symmetric **AES-256-GCM** key is generated by the sender, encrypted with the recipient's public key, and sent to establish a secure channel.

4.  **Message Encryption & Signing**: Messages are encrypted with the shared AES key. The resulting ciphertext is then signed with the sender's RSA private key to prove authenticity to the recipient.

#### **4. API & Server Logic**

##### **Authentication (JWT)**

A JWT is **required** for a client to interact with the API. It separates the concern of *server access* from *message privacy*.

*   **Purpose**:
    *   **Rate Limiting & Anti-Spam**: Prevents unauthorized or malicious clients from overwhelming the server with requests.
    *   **Authorized Fetching**: Ensures that when a user requests their messages, the server returns only the messages addressed to them. The JWT securely identifies the user making the request.
    *   **Server-Level Authentication**: The JWT proves to the server that the entity uploading data is an authenticated user, distinct from the E2EE signature which proves message origin to the *recipient*.
*   **Alternative (Not Recommended)**: Signing every API request is computationally expensive for the server, non-standard, and unnecessarily exposes the long-term private key to more frequent use.

##### **API Endpoints:**

1.  `POST /auth/register-challenge`
    *   **Request**: `{ "alias": "string" }`
    *   **Response**: `{ "nonce": "string" }`

2.  `POST /auth/register-complete`
    *   **Request**: `{ "alias": "string", "publicKey": "string", "signedNonce": "string" }`
    *   **Action**: Server verifies `signedNonce` with `publicKey`. If valid, creates the user record.
    *   **Response**: `201 Created`

3.  `POST /auth/login-challenge`
    *   **Request**: `{ "alias": "string" }`
    *   **Response**: `{ "challenge": "string" }` (A server-generated nonce or timestamp).

4.  `POST /auth/login-complete`
    *   **Request**: `{ "alias": "string", "signedChallenge": "string" }`
    *   **Action**: Server verifies the signature against the user's stored public key. If valid, it issues a short-lived JWT (e.g., 1-hour expiry) containing the user's `alias` or a `userId`.
    *   **Response**: `{ "token": "jwt-token-here" }`

5.  `GET /keys/:alias`
    *   **Auth**: JWT required.
    *   **Response**: `{ "alias": "string", "publicKey": "string" }`

6.  `POST /messages`
    *   **Auth**: JWT required.
    *   **Request**:
        ```json
        {
          "recipientAlias": "string",
          "type": "KEY_EXCHANGE | TEXT | FILE",
          // For TEXT/KEY_EXCHANGE: Base64 encoding of IV + Ciphertext + Auth Tag
          // For FILE: encrypted file content (server stores separately, returns file reference)
          "encryptedMessage": "string",
          // RSA signature of the encryptedMessage string
          "signature": "string"
        }
        ```
    *   **FILE Type Handling**: Server generates a UUID, stores encrypted content in `files` table, replaces `encryptedMessage` with `{"fileId": "uuid"}` before storing in messages.
    *   **Response**: `202 Accepted`

7.  `GET /messages?since=<id>`
    *   **Auth**: JWT required.
    *   **Action**: Long polling to fetch new messages where JWT subject is the recipient. Uses **tick counter** (message ID) instead of timestamp for reliable deduplication.
    *   **Query**: `since` is an integer message ID. Returns all messages with `id > since`.
    *   **Response**: `[ ...message objects with "id" field... ]`

8.  `GET /files/:fileId`
    *   **Auth**: JWT required.
    *   **Action**: Retrieves encrypted file content. Only accessible by the sender or recipient of the file.
    *   **Response**: `{ "fileId": "string", "encryptedContent": "string" }`

#### **5. Data Models (SQLite)**

**Users Table:**
```sql
CREATE TABLE users (
    alias TEXT PRIMARY KEY,
    publicKey TEXT NOT NULL
);
```

**Messages Table:**
```sql
CREATE TABLE messages (
    id INTEGER PRIMARY KEY AUTOINCREMENT,  -- Tick counter for polling
    recipientAlias TEXT NOT NULL,
    senderAlias TEXT NOT NULL,
    type TEXT NOT NULL,                    -- KEY_EXCHANGE | TEXT | FILE
    encryptedMessage TEXT NOT NULL,        -- For FILE: {"fileId": "uuid"}
    signature TEXT NOT NULL,
    serverTimestamp TEXT NOT NULL
);
```

**Files Table:**
```sql
CREATE TABLE files (
    id TEXT PRIMARY KEY,                   -- UUID
    messageId INTEGER NOT NULL,
    senderAlias TEXT NOT NULL,
    recipientAlias TEXT NOT NULL,
    encryptedContent TEXT NOT NULL,        -- Encrypted file data
    FOREIGN KEY (messageId) REFERENCES messages (id)
);
```

#### **6. Client-Side Cryptography Implementation Details**

*   **AES-GCM Security**:
    *   **IV (Initialization Vector)**: The IV **does not need to be secret** and is sent in plaintext (packaged within `encryptedMessage`). Its purpose is to ensure identical plaintext messages produce unique ciphertexts.
    *   A new **12-byte (96-bit)** IV **must** be generated for every message using a Cryptographically Secure Pseudo-Random Number Generator (CSPRNG).
    *   **CRITICAL**: Reusing an `(IV, Key)` pair for two different messages destroys the security of the channel.
    *   **Authentication Tag**: AES-GCM produces a **16-byte (128-bit)** tag. This is not optional. It allows the decrypting party to verify that the message has not been tampered with.
*   **Packaging & Parsing `encryptedMessage`**:
    *   **To Encrypt**:
        1.  Generate a random 12-byte `IV`.
        2.  Encrypt the plaintext using `AES-GCM` with the symmetric key and `IV`. This produces `Ciphertext` and a 16-byte `Tag`.
        3.  Concatenate the raw bytes: `IV` + `Ciphertext` + `Tag`.
        4.  Base64 encode the final concatenated byte array. This is your `encryptedMessage`.
    *   **To Decrypt**:
        1.  Base64 decode the `encryptedMessage` string into a byte array.
        2.  Slice the first 12 bytes to get the `IV`.
        3.  Slice the last 16 bytes to get the `Tag`.
        4.  The bytes in the middle are the `Ciphertext`.
        5.  Run the AES-GCM decryption function with these components. The function will automatically use the `Tag` to verify integrity. If verification fails, the function will throw an error. In this case, the entire message **must be discarded**. The ciphertext is considered untrusted, and no partial data should ever be displayed.
*   **File Handling**:
    *   Files are encrypted client-side using the same AES-GCM session key as text messages.
    *   The `encryptedMessage` for FILE type contains the full encrypted file content.
    *   After receiving a FILE message, the client uses the `fileId` from the message body to fetch content via `GET /files/:fileId`.

#### **7. Security & Privacy Considerations**

*   **Metadata Leakage**: This design protects message *content*, but the server still knows *who* is talking to *whom* and *when* (metadata). Protecting this would require a more complex system like an onion routing network (e.g., Tor).
*   **Forward Secrecy**: The current design uses long-lived session keys (`SYM_AB`). If a session key is ever compromised, all past messages encrypted with that key could be decrypted. A more advanced system would implement "Forward Secrecy" using a protocol like Signal's Double Ratchet, where keys are constantly updated. This is out of scope for v1.
*   **File Storage Security**: Files are stored in the database (not filesystem), keyed by server-generated UUIDs. The `/files/:fileId` endpoint enforces authorization - only sender or recipient can access.
